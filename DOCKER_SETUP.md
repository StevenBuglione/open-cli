# Local Authentik Setup for `ocli` + `oclird`

The supported local auth workflow reuses the repo's validated Authentik reference stack under `product-tests/authentik/` and renders:

- `config/.cli.authentik.local.json` for the Docker-hosted `oclird` daemon
- `.cli.local.json` for workload-style `oauthClient` auth from local `ocli`
- `.browser-login/.cli.json` for operator `browserLogin` auth from local `ocli`
- `.authentik.local.env` for the confidential client credentials used by the workload flow
- `.authentik.docker.env` for the Docker Compose `OCLIRD_CONFIG_PATH` override

## What talks to what

Authentik is the identity provider in this setup. It issues the token, but it is not the runtime API server.

`ocli` still talks to `oclird`, and `oclird` is still required because it:

- loads the config and effective catalog
- validates the Authentik-issued token
- applies the authorization envelope and scope filtering
- exposes `/v1/runtime/info` and `/v1/auth/browser-config`
- executes tools against the configured upstream services

The high-level flow is:

```text
ocli -> oclird -> validates Authentik token -> returns filtered catalog / executes tools
```

## Quick start

### Hosted runtime + workload flow (`oauthClient`)

From the repo root:

```bash
./scripts/setup-authentik.sh
source ./.authentik.local.env
source ./.authentik.docker.env
docker compose up -d oclird
```

In a second shell:

```bash
source ./.authentik.local.env
go run ./cmd/ocli --config ./.cli.local.json catalog list --format pretty
```

### Browser login note

The setup script still writes `.browser-login/.cli.json`, but the hosted daemon split above is currently verified for workload-style `oauthClient` auth.

Browser login still needs a daemon config whose runtime auth issuer matches the public browser provider. That browser-specific hosted daemon config is not auto-generated by this split yet.

What the setup script does:

- starts the Authentik reference stack with `docker compose -f product-tests/authentik/docker-compose.yml up -d`
- bootstraps a confidential Authentik provider/application for workload auth through `manage.py shell`
- bootstraps a public Authentik provider/application for browser PKCE auth
- emits a runtime-compatible `scope` claim from `token.scope`
- renders `config/.cli.authentik.local.json` with the hosted runtime catalog plus schema-valid `issuer` and `jwksURL` fields
- renders `.cli.local.json` as a minimal remote client config with only runtime URL and OAuth client credentials settings
- renders `.browser-login/.cli.json` as a minimal browser-login client config for future browser-matched daemon wiring
- writes `.authentik.local.env` with `OAS_REMOTE_CLIENT_ID` and `OAS_REMOTE_CLIENT_SECRET`
- writes `.authentik.docker.env` with the `OCLIRD_CONFIG_PATH` override for Docker Compose

## Defaults

- Authentik UI/API: `http://127.0.0.1:9100`
- Runtime URL in generated config: `http://127.0.0.1:8765`
- Runtime audience: `oclird`
- Generated runtime scope: `bundle:testapi`
- OpenAPI source: `./product-tests/testdata/openapi/testapi.openapi.yaml`
- Browser callback port: `8787` by default, configurable with `OCLI_BROWSER_CALLBACK_PORT`

Catalog inspection works without any upstream service. To exercise execution against the checked-in test API, also start:

```bash
cd product-tests && make services-up
```

## Useful overrides

You can customize the generated local config with environment variables before running the setup script:

```bash
OCLI_RUNTIME_SERVICE_ID=petstore \
OCLI_OPENAPI_URI=https://petstore3.swagger.io/api/v3/openapi.json \
./scripts/setup-authentik.sh
```

To request multiple bundle scopes and expose the same OpenAPI fixture under multiple local service IDs, add `OCLI_RUNTIME_EXTRA_SERVICE_IDS`:

```bash
OCLI_RUNTIME_EXTRA_SERVICE_IDS="testapi-admin" \
./scripts/setup-authentik.sh
```

That generates configs whose requested scopes include both:

- `bundle:testapi`
- `bundle:testapi-admin`

and whose effective catalog can expose both `testapi:*` and `testapi-admin:*` tools when the token carries both scopes.

Other supported overrides:

- `AUTHENTIK_BASE_URL`
- `OCLI_RUNTIME_URL`
- `OCLI_RUNTIME_AUDIENCE`
- `OCLI_RUNTIME_EXTRA_SERVICE_IDS`
- `OCLI_RUNTIME_EXTRA_SCOPES`
- `OCLI_AUTHENTIK_CLIENT_SLUG`
- `OCLI_AUTHENTIK_BROWSER_CLIENT_SLUG`
- `OCLI_LOCAL_CONFIG_PATH`
- `OCLI_DAEMON_CONFIG_PATH`
- `OCLI_BROWSER_CONFIG_PATH`
- `OCLI_LOCAL_ENV_PATH`
- `OCLI_DOCKER_ENV_PATH`
- `OCLI_BROWSER_CALLBACK_PORT`

## Generated files

- `config/.cli.authentik.local.json`: hosted `oclird` daemon config
- `.cli.local.json`: minimal confidential-client workload config
- `.browser-login/.cli.json`: minimal public-client browser-login config
- `.authentik.local.env`: local exports for `OAS_REMOTE_CLIENT_ID` and `OAS_REMOTE_CLIENT_SECRET`
- `.authentik.docker.env`: local export for `OCLIRD_CONFIG_PATH`

These are local artifacts and should not be committed.

## Cleanup

Stop the Authentik reference stack:

```bash
cd product-tests && make authentik-down
```

Stop the test API stack if you started it:

```bash
cd product-tests && make services-down
```
